# Лабораторна робота 4: Аналіз синхронної та асинхронної комунікації

## Що змінилося порівняно з лабораторною 3?

У лабораторній 3 `SubscribeCommandHandler` безпосередньо залежав від `IMailer` і викликав `mailer.sendMail()` прямо всередині своєї основної операції. Формально DIP дотримувався (через інтерфейс), але логічно відправка пошти була частиною команди, а не побічним ефектом.

У лабораторній 4 комунікацію з поштовим компонентом винесено окремо двома способами:

- **Синхронний варіант** (`SubscribeCommandHandler`): після збереження підписки handler викликає `INotificationService.onSubscriptionConfirmed(event)` напряму. Час відповіді = час основної операції + час відправки листа (~200–800 мс для SMTP).
- **Асинхронний варіант** (`SubscribeCommandHandlerAsync`): handler публікує `SubscriptionConfirmedEvent` в `InMemoryEventBus` і повертається негайно. `EmailNotificationService` підписаний на цю подію і обробляє її поза основним потоком виконання через `setImmediate`.

Новий компонент `EmailNotificationService` тепер має власний модуль (`src/application/notifications/`), чіткий контракт (`INotificationService`) і не знає нічого про HTTP, команди чи домен — лише реагує на події.

## Які побічні операції виділено і чому?

**Відправка підтверджувального email** — після успішного збереження підписки. Це класична побічна операція: вона не є частиною бізнес-інваріанту (підписка вже збережена в БД незалежно від того, чи дійшов лист), але є важливою з точки зору UX. Збій SMTP не має відкочувати транзакцію підписки.

**Відправка нотифікацій про реліз** — після того, як scanner виявляє новий тег. Це також побічний ефект сканування: основна задача scanner'а — оновити `last_seen_tag` в БД, а розсилка є наслідком цього факту.

## Порівняння підходів

### Час відповіді API

- **Синхронно**: POST `/subscribe` чекає на завершення SMTP-запиту. При нормальній роботі Gmail SMTP відповідає за ~200–500 мс, тобто загальний час відповіді зростає на цей час.
- **Асинхронно**: POST `/subscribe` повертає 200 одразу після `saveSubscription()`. Лист відправляється в наступній мікрозадачі (`setImmediate`). Час відповіді API — лише час core операції (~20–50 мс з БД).

### Поведінка при збоях

- **Синхронно**: якщо SMTP упав — клієнт отримає 500, навіть якщо підписка успішно збережена в БД. Це оманливо: підписка є, але відповідь каже про помилку. Рішення — ігнорувати помилку notification (`try/catch` навколо виклику), але тоді клієнт не знає, чи дійшов лист.
- **Асинхронно**: SMTP failure логується в консоль, але клієнт отримує 200. Підписка збережена — це головне. Втрата події при падінні процесу — можливий недолік in-process bus (немає persistence).

### Зв'язаність між компонентами

- **Синхронно**: `SubscribeCommandHandler` знає про `INotificationService` і залежить від нього. Якщо додати ще одну побічну операцію (аудит, аналітика) — треба додати ще один параметр у конструктор.
- **Асинхронно**: handler знає тільки про `IEventBus`. Новий підписник на `SubscriptionConfirmed` додається без жодних змін у handler — він просто реєструється в `composition.root.ts`. Це відкритий/закритий принцип на практиці.

### Складність реалізації та тестування

- **Синхронно**: простіше реалізувати і тестувати — звичайний mock `INotificationService`, перевірити що метод викликано.
- **Асинхронно**: тестування потребує очікування `setImmediate` (`await new Promise(res => setImmediate(res))`). Реалізація `InMemoryEventBus` проста, але потребує розуміння event loop. Для production потрібен надійніший bus (Redis Streams, RabbitMQ) з persistence та retry.

## Який підхід для production?

**Асинхронний** — але не in-process. У production я б використав **Redis Streams** або **RabbitMQ**:

1. Підписка зберігається в БД у транзакції.
2. Подія публікується в чергу (бажано в тій же транзакції через outbox pattern).
3. Окремий worker читає з черги і відправляє листи з retry логікою.

Це дає: швидкий API, ізоляцію збоїв, можливість retry при SMTP-помилках, горизонтальне масштабування workers.

In-process `InMemoryEventBus` підходить для навчальних цілей та невеликих систем, але при рестарті процесу непроцесовані події губляться.

## Ідемпотентність обробників

Поточна реалізація **не є ідемпотентною**: якщо `SubscriptionConfirmedEvent` доставиться двічі (наприклад, при retry), користувач отримає два листи з одним токеном.

Для забезпечення ідемпотентності потрібно:
- Зберігати `event.occurredAt` + хеш події як `processed_event_id` в БД.
- Перед відправкою перевіряти чи ця подія вже оброблена.
- Альтернатива — токен підтвердження одноразовий: після використання видаляється, тому повторна відправка листа не дає змогу підтвердити підписку двічі.

## Висновок

Асинхронна комунікація через event bus є більш правильним архітектурним рішенням для побічних ефектів типу email-нотифікацій: вона знижує зв'язаність, прискорює відповідь API та ізолює збої зовнішніх сервісів. Синхронний підхід простіший у реалізації, але жорстко прив'язує час відповіді і поведінку при збоях до надійності SMTP-сервера. Для production-системи рекомендовано асинхронний підхід з зовнішнім брокером та outbox pattern.

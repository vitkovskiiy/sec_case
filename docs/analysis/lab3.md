# Лабораторна робота 3: Аналіз CQS (Command-Query Separation)

## Що змінилося в структурі проєкту порівняно з лабораторною 2?

У лабораторній 2 кожен юзкейс реалізовувався через окремий ApplicationService (`SubscribeService`, `FindService`, `UnsubscribeService`, `TokenService`). Контролери залежали від цих сервісів і безпосередньо викликали їхні методи.

У лабораторній 3 Application Layer було розділено на дві незалежні гілки:

- **Commands** (`src/application/commands/`) — об'єкти-наміри та їхні обробники для кожної операції запису:
  - `SubscribeCommand` + `SubscribeCommandHandler`
  - `ConfirmSubscriptionCommand` + `ConfirmSubscriptionCommandHandler`
  - `UnsubscribeCommand` + `UnsubscribeCommandHandler`

- **Queries** (`src/application/queries/`) — об'єкти запитів та їхні обробники для читання:
  - `GetSubscriptionsByEmailQuery` + `GetSubscriptionsByEmailQueryHandler`

- **Read Models** (`src/application/read-models/`) — окремі DTO для відповідей на запити (`SubscriptionReadModel`), відокремлені від доменних сутностей.

- **Read Repository** (`ISubscriptionQueryRepository`) — окремий інтерфейс для читання, що не змішується з write-репозиторієм.

Контролери стали суто «тонкими»: вони лише маппять HTTP-запит у Command або Query і передають його відповідному Handler. Жодної логіки в контролерах немає.

## Переваги CQS: чим допомогло розділення?

**Явна семантика коду.** Назва класу одразу говорить, чи це дія (`SubscribeCommand`) чи запит (`GetSubscriptionsByEmailQuery`). Розробник не мусить читати реалізацію, щоб зрозуміти, чи мутує цей код стан.

**Різні вимоги — різний код.** Query Handler може напряму читати з БД, оминаючи доменний шар, і повертати `SubscriptionReadModel`, оптимізований під клієнта. Command Handler навпаки — обов'язково проходить через домен і перевіряє інваріанти. Раніше цей поділ був неявним.

**Простіше тестування команд.** Unit-тести команд тестують суто поведінку: "якщо репо не існує — кинути помилку", "якщо вже підписаний — кинути помилку". Всі зовнішні залежності замінюються на моки. Тести швидкі, без БД, без HTTP.

**Незалежна еволюція.** Додати нову команду — це створити два нових файли (`XyzCommand` + `XyzCommandHandler`) без жодних змін у існуючому коді. У лаб. 2 довелося б або додавати метод у наявний сервіс, або створювати новий сервіс і перереєстровувати залежності.

**Оптимізація читань.** Query Handler та його репозиторій (`ISubscriptionQueryRepository`) можуть бути оптимізовані окремо: cache, materialized views, денормалізовані таблиці — все це не торкається write-side.

## Недоліки CQS: що ускладнилося?

**Більше файлів і класів.** Один юзкейс тепер складається щонайменше з двох класів (`Command` + `Handler`). Для невеликого проєкту це додає boilerplate без видимої вигоди.

**Немає вбудованої диспетчеризації.** У повноцінному CQRS є `CommandBus` / `QueryBus`, які автоматично направляють команду до потрібного хендлера. У нашому варіанті ця логіка вшита в `composition.root.ts` вручну — при зростанні кількості команд цей файл розростатиметься.

**Розрив між записом і читанням.** Після виконання команди контролер не може повернути оновлений стан (команди не повертають дані). Якщо клієнту потрібна відповідь з оновленими даними після POST, доводиться або робити окремий GET, або порушувати принцип (повертати ID).

**Складніша навігація.** Новий розробник у команді мусить знати, де шукати логіку: команди — в `commands/`, запити — в `queries/`, домен — в `domain/`. У простому сервісі все було в одному місці.

## Чим підхід з Command / Query Handler відрізняється від Service, який робить все сам?

| Критерій | ApplicationService (лаб. 2) | Command/Query Handler (лаб. 3) |
|---|---|---|
| Розмір | Один сервіс, кілька методів | Один клас — один обробник |
| Відповідальність | Читання + запис разом | Чітко розділені |
| Залежності | Всі в одному конструкторі | Тільки ті, що потрібні |
| Розширюваність | Додати метод у клас | Додати новий клас |
| Тестування | Тестуємо клас з усіма методами | Тестуємо кожен хендлер ізольовано |

Сервіс — це "все в одному". Handler — це принцип єдиної відповідальності, доведений до конкретного юзкейсу.

## Як CQS впливає на розширюваність?

Додати нову команду (наприклад, `PauseSubscriptionCommand`) означає:
1. Створити `PauseSubscriptionCommand.ts` — DTO з даними
2. Створити `PauseSubscriptionCommandHandler.ts` — логіка
3. Зареєструвати в `composition.root.ts`
4. Додати маппінг у контролері

Жодного існуючого коду не чіпаємо. У лаб. 2 довелося б додати метод у `SubscribeService`, що потенційно ламає існуючі тести цього сервісу.

## Чи відрізняється структура даних, яку повертає Query, від доменної моделі? Чому це важливо?

Так, відрізняється. Доменна модель `Subscription` містить лише `email` та `repo` — поля, необхідні для забезпечення бізнес-інваріантів. Натомість `SubscriptionReadModel` містить `email`, `repo`, `confirmed`, `last_seen_tag` — тобто дані, оптимізовані під потреби UI.

Це важливо з кількох причин:

- **Доменна модель захищена.** Вона не "протікає" у клієнтський код. Якщо домен зміниться — контракт API не зміниться автоматично.
- **Read Model може включати дані з кількох таблиць.** Query робить JOIN напряму в SQL і повертає плоску структуру — без зайвих об'єктів і маппінгу через доменний шар.
- **Версіонування.** Read Models можна змінювати незалежно від домену під потреби різних клієнтів (мобільний додаток може отримувати скорочену версію, адмін-панель — повну).

## Висновок

CQS є природнім кроком після Clean Architecture: він формалізує те, що раніше було неявним розділенням у сервісах. Для нашого проєкту `github-release-notifier` він чітко відокремив операції підписки/відписки/підтвердження (команди) від операції перегляду підписок (запит). Тести стали більш сфокусованими: unit-тести команд перевіряють доменну логіку без БД, інтеграційні тести запитів перевіряють HTTP-відповідь та форму Read Model. Головний недолік — збільшення кількості файлів, що виправдано лише при подальшому зростанні проєкту.
